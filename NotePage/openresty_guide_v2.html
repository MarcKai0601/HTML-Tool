Chapter 01
OpenResty 是什麼？
在學配置之前，先搞清楚它解決的問題——以及為什麼不用純 Nginx 就好
本章學完你會知道
OpenResty 跟 Nginx 的關係是什麼
它比純 Nginx 多了什麼能力
在架構中它扮演什麼角色
什麼情況下應該選它
OpenResty = Nginx + LuaJIT
一句話解釋：OpenResty 就是在 Nginx 裡裝了一個 Lua 執行引擎（LuaJIT），讓你可以在 Nginx 處理請求的過程中插入自訂邏輯——驗證 Token、查 Redis、做路由判斷，而且這些操作是非阻塞的，不會影響 Nginx 同時處理其他請求。
Nginx（基底）
高效能 HTTP Server，事件驅動模型，處理靜態檔案、反向代理。但設定檔是靜態的，if 能力有限，沒辦法寫複雜邏輯。
靜態配置反向代理
LuaJIT（擴充）
高效能 Lua 直譯器，效能接近 C。搭配協程（Coroutine）實現非阻塞 I/O——寫起來像同步，執行起來不阻塞。
動態邏輯Coroutine
lua-resty-* 模組
豐富的官方與社群模組：Redis、MySQL、HTTP Client、JWT、Session 等，全部都是非阻塞實作。
resty.redisresty.jwt

OpenResty vs 純 Nginx vs 後端 Gateway
方案動態邏輯效能適合場景
純 Nginx 幾乎沒有，if 有限制 極高 純靜態檔案、簡單反向代理
OpenResty Lua 完整邏輯能力 極高 高流量 API 入口、Gateway
Kong / APISIX Plugin 生態，開箱即用 高 標準化 Gateway，不想自己寫 Lua
Spring Cloud Gateway Java 完整生態 中（JVM 開銷） Java 微服務生態、已有 Spring
Kong 和 APISIX 底層就是 OpenResty。理解 OpenResty 等於理解這兩個主流 API Gateway 的運作原理。
本章小結
OpenResty = Nginx + LuaJIT，在 Nginx 請求生命週期中插入 Lua 邏輯
非阻塞 I/O：Lua 查 Redis 時不會卡住其他請求
在架構中作為統一入口，負責驗證、限流、路由，後端服務只處理業務邏輯
Kong / APISIX 底層皆基於 OpenResty

安裝
CentOS / RHEL

yum install -y yum-utils

  https://openresty.org/package/centos/openresty.repo
yum install -y openresty

新增 repo 並安裝
apt install -y software-properties-common
add-apt-repository -y ppa:openresty/ppa
apt update
apt install -y openresty
啟動與管理
1
確認版本（確認安裝成功）
openresty -v
會看到類似 nginx version: openresty/1.25.3.1
2
語法檢查（修改配置後先檢查，不重啟）
openresty -t
若看到 syntax is ok 才繼續下一步。養成習慣：改完先 -t 再 reload。
3
啟動
sudo openresty
啟動後用 curl http://localhost 確認，看到 Welcome to OpenResty! 代表成功。
4
Graceful Reload（生產環境修改配置後用這個）
sudo openresty -s reload
不中斷現有連線，新配置對新請求生效。生產環境永遠優先用 reload，不要 stop + 重啟。
5
停止
sudo openresty -s stop
reload vs stop+重啟的差別：reload 讓舊的 Worker 處理完現有連線後才退出，新 Worker 用新配置接新請求。stop 是立即中斷所有連線。生產環境幾乎只用 reload。

# 確認監聽的 port
ss -tlnp | grep nginx

# 找配置檔位置
find / -name nginx.conf 2>/dev/null | grep openresty

# 預設路徑
/usr/local/openresty/nginx/conf/nginx.conf

完整 nginx.conf 範本
nginx.conf（含重要說明）
user  root;
worker_processes  auto;          # 建議設 auto

events {
    worker_connections  102400;  # 每個 Worker 最大連線數
}

http {
    # ── Buffer 設定 ──────────────────────────────────
    client_max_body_size 200M;
    client_header_buffer_size 256k;
    proxy_buffers 128 32k;
    proxy_buffer_size 64k;

    include  mime.types;
    default_type  application/octet-stream;

    # 隱藏 Server header（避免暴露版本資訊）
    more_set_headers 'Server: -';

    # ── Log 格式 ─────────────────────────────────────
    log_format main '[$request_time] | $time_local | $remote_addr'
                    ' | $http_host | $request | $status'
                    ' | "$upstream_response_time" | $upstream_addr';

    # ── Lua 設定 ─────────────────────────────────────
    lua_shared_dict shared_data 512m;  # 跨 Worker 共享記憶體
    lua_code_cache on;                 # 生產環境必須 on
    init_by_lua_file /usr/local/openresty/nginx/lua/init.lua;

    # ── 傳遞真實 IP ───────────────────────────────────
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

    sendfile on;
    keepalive_timeout 65;

    # ── 引入子配置 ────────────────────────────────────
    include /usr/local/openresty/nginx/conf/blockip.conf;
    include /usr/local/openresty/nginx/conf/logable.conf;
    include /usr/local/openresty/nginx/conf/upstream/*.conf;
    include /usr/local/openresty/nginx/conf/conf.d/*.conf;
}
location 匹配規則與優先順序
語法說明優先級
= /exact完全匹配，只有這條路徑符合最高
^~ /prefix前綴匹配，找到後不再繼續比對正則高
~ /regex正則匹配，區分大小寫中
~* /regex正則匹配，不區分大小寫中
/prefix一般前綴，最長的優先低
proxy_pass 結尾斜線是常見坑：
proxy_pass http://backend/api/ → location /app/ 的請求 /app/user 會被轉成 /api/user（路徑替換）
proxy_pass http://backend → 保留完整路徑 /app/user 不變
不確定時建議用無斜線版本，行為更直覺。
本章小結
conf.d/ 目錄每個 domain 一個 .conf 檔，用 include 全部載入
lua_code_cache on 生產環境必開，否則每次請求重新編譯
location 優先順序：= > ^~ > ~ > ~* > 一般前綴
proxy_pass 結尾斜線會影響路徑拼接，務必測試確認

最基本的反向代理
Client 的請求進來後，Nginx 轉發給後端服務（Spring Boot / Go 等），再把回應回傳給 Client。後端服務不直接對外，只有 Nginx 的 port 對外開放。
最基本範例：把 /api/ 轉發給後端 8080
server {
    listen 80;
    server_name localhost;

    location /api/ {
        include /usr/local/openresty/nginx/conf/proxy.conf;
        proxy_pass http://127.0.0.1:8080;
    }
}
proxy.conf — 統一管理代理 Header
每次用 proxy_pass 都 include 這個檔，確保後端永遠能拿到正確的 Client IP 和協議資訊。
proxy.conf
proxy_http_version 1.1;
proxy_set_header Connection "";           # 啟用 keep-alive 到 upstream
proxy_set_header Host $host;
proxy_set_header X-Real-IP $remote_addr;
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
proxy_set_header X-Forwarded-Proto $scheme;
為什麼要設 HTTP 1.1 + Connection ""？預設 Nginx 對 upstream 用 HTTP 1.0，每次請求結束連線就關掉。改成 1.1 + keepalive 讓連線可以重複使用，避免頻繁建立 TCP 三次握手，高流量下效能差異明顯。
Upstream — 多台後端負載平衡
upstream/api.conf
upstream backend_api {
    least_conn;   # 負載平衡演算法（見下表）

    server 10.20.35.152:8080 weight=5 max_fails=3 fail_timeout=10s;
    server 10.20.35.120:8080 weight=5 max_fails=3 fail_timeout=10s;

    keepalive 64;  # 維持 64 條長連線到後端
}
server block 中引用 upstream
location /api/ {
    include /usr/local/openresty/nginx/conf/proxy.conf;
    proxy_pass http://backend_api;   # 直接用 upstream 名稱
}
演算法說明適合場景
round-robin（預設）輪流分配，每台輪一次後端效能相近
weight=N按比例，weight 越高收越多請求機器規格不同
least_conn選當前連線數最少的推薦：長連線 API
ip_hash同一 IP 固定打同一台需要 session 黏性（不推薦，應改用 Redis session）
Timeout 設定
Timeout 設定（放在 location 或 server 內）
proxy_connect_timeout  5s;    # 建立連線最多等 5 秒
proxy_send_timeout     30s;   # 送出請求最多等 30 秒
proxy_read_timeout     30s;   # 等待後端回應最多 30 秒

# 後端出錯時自動轉到下一台
proxy_next_upstream error timeout http_500 http_502;
max_fails + fail_timeout 被動健康檢查：max_fails=3 代表 3 次失敗後，該機器被標記為不可用，fail_timeout=10s 後再試。這是 Nginx 內建的被動式健康檢查，不需要額外模組。
本章小結
proxy_pass 轉發流量，proxy.conf 統一管理 header，每次都 include
多台機器用 upstream block，keepalive 維持長連線到後端
長連線 API 推薦 least_conn，避免某台機器堆積慢請求
max_fails + fail_timeout 是 Nginx 內建被動健康檢查，不需額外模組

三、完整 HTTPS server block
conf.d/api.example.com.conf
# ── HTTP 強制轉 HTTPS ─────────────────────────────
server {
    listen 80;
    server_name api.example.com;
    return 301 https://$host$request_uri;
}

# ── HTTPS 主配置 ──────────────────────────────────
server {
    listen 443 ssl;
    server_name api.example.com;

    # 憑證
    ssl_certificate     /etc/letsencrypt/live/api.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/api.example.com/privkey.pem;

    # SSL 安全設定
    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_ciphers         HIGH:!aNULL:!MD5;
    ssl_prefer_server_ciphers on;
    ssl_session_cache   shared:SSL:10m;
    ssl_session_timeout 10m;

    # 安全 Header
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header X-Frame-Options DENY;
    add_header X-Content-Type-Options nosniff;

    access_log  /var/log/nginx/api.access.log  main;
    error_log   /var/log/nginx/api.error.log;

    location / {
        include /usr/local/openresty/nginx/conf/proxy.conf;
        proxy_pass http://backend_api;
    }

    error_page 500 502 503 504 /50x.html;
    location = /50x.html {
        root /usr/local/openresty/nginx/html;
    }
}
四、多 domain 一台機器（Virtual Host）
每個 domain 一個 .conf 檔放在 conf.d/，nginx.conf 最後 include conf.d/*.conf 全部載入。TLS 握手時 Client 會透過 SNI 告知要連哪個 domain，Nginx 自動選對應的憑證和 server block。
# conf.d/
# ├── api.example.com.conf     → proxy_pass http://backend_api
# ├── admin.example.com.conf   → proxy_pass http://backend_admin（限 IP）
# └── static.example.com.conf  → root /var/www/static（靜態檔案直接 serve）

# 靜態資源直接用 root，不要 proxy，效能差距 10x 以上
server {
    listen 443 ssl;
    server_name static.example.com;
    # ... ssl 設定 ...
    root /var/www/static;
    location / {
        try_files $uri $uri/ =404;
        add_header Cache-Control "public, max-age=31536000";
    }
}

DNS A Record 指向 Server IP，多個 subdomain 可以指向同一台
certbot 申請免費憑證，加 crontab 自動每月續期
HTTP server block 只做 301 轉 HTTPS，邏輯都在 443 server block
多 domain 每個建一個 .conf 檔，用 include conf.d/*.conf 全部載入

Lua 基本語法
變數宣告
-- 區域變數（推薦）
local name = "Kai"
local age = 29
local active = true

-- 全域變數（盡量避免）
greeting = "hello"

-- 字串拼接用 .. （不是 +）
local msg = "Hello, " .. name
條件判斷
local score = 80

if score >= 90 then
    ngx.say("A")
elseif score >= 60 then
    ngx.say("B")
else
    ngx.say("F")
end

-- 不等於用 ~= （不是 !=）
if name ~= "admin" then
    ngx.exit(403)
end
迴圈
-- 數字迴圈
for i = 1, 5 do
    ngx.say(i)
end

-- 遍歷 table（類似 for-each）
local fruits = {"apple", "banana", "mango"}
for i, v in ipairs(fruits) do
    ngx.say(i, ": ", v)
end
函式
-- 定義函式
local function greet(name)
    return "Hello, " .. name
end

-- 呼叫
local msg = greet("Kai")
ngx.say(msg)

-- 函式可以回傳多個值
local function minmax(a, b)
    return math.min(a,b), math.max(a,b)
end
local lo, hi = minmax(3, 7)
Lua Table（最重要的資料結構）
Table 是 Lua 唯一的資料結構，可以同時當 Array、HashMap、物件使用。OpenResty 中幾乎所有 API 回傳的都是 table。
當 Array 用
local fruits = {"apple", "banana"}

-- Lua index 從 1 開始（不是 0）
ngx.say(fruits[1])   -- apple
ngx.say(#fruits)     -- 2（長度）

-- 新增
fruits[3] = "mango"
table.insert(fruits, "grape")
當 HashMap 用
local user = {
    name = "Kai",
    age = 29,
    role = "admin"
}

ngx.say(user.name)       -- Kai
ngx.say(user["role"])    -- admin

-- 遍歷 HashMap
for k, v in pairs(user) do
    ngx.say(k, " = ", v)
end

lua/utils.lua（自訂模組）
local M = {}   -- M 是這個模組的 table

function M.greet(name)
    return "Hello, " .. name
end

function M.is_admin(role)
    return role == "admin"
end

return M       -- 模組必須 return

-- 載入自訂模組
local utils = require "utils"

-- 載入內建模組
local cjson = require "cjson"
local jwt   = require "resty.jwt"
local redis = require "resty.redis"

請求生命週期（Phase）
一個請求進來，Nginx 會依序經過以下階段，每個階段都可以用不同的 Lua hook 插入邏輯。
init_by_lua
Worker 啟動時執行一次。初始化設定、載入模組。不是每個 request 都走。
rewrite_by_lua
修改 URI、做 URL 轉向、動態路由（灰度發布）
access_by_lua
驗證 Token、IP 白名單、Rate Limit。驗證失敗直接 exit，不進後端。
content_by_lua
直接回傳內容（快取命中、健康檢查）。與 proxy_pass 互斥。
header_filter_by_lua
修改 response header（加 CORS、移除敏感 header）
log_by_lua
記錄 log、埋監控指標。不影響 response，用於後置處理。
實戰一：JWT 驗證（access 階段）
lua/jwt_auth.lua
local jwt   = require "resty.jwt"
local cjson = require "cjson"

-- 讀取 Authorization header（注意：全小寫）
local auth_header = ngx.var.http_authorization
if not auth_header then
    ngx.status = 401
    ngx.header["Content-Type"] = "application/json"
    ngx.say(cjson.encode({ error = "Missing Authorization header" }))
    return ngx.exit(401)
end

-- 取出 Bearer token
local token = auth_header:match("Bearer%s+(.+)")
if not token then
    ngx.status = 401
    ngx.say(cjson.encode({ error = "Invalid format. Use: Bearer " }))
    return ngx.exit(401)
end

-- 驗證 JWT（secret 從 shared_dict 讀，不要 hardcode）
local config = ngx.shared.shared_data
local secret = config:get("jwt_secret")

local jwt_obj = jwt:verify(secret, token)
if not jwt_obj.verified then
    ngx.status = 403
    ngx.say(cjson.encode({ error = "Invalid token", reason = jwt_obj.reason }))
    return ngx.exit(403)
end

-- 把 user_id 注入 header，後端直接信任，不用再次驗證
ngx.req.set_header("X-User-Id", jwt_obj.payload.sub)
ngx.req.set_header("X-User-Role", jwt_obj.payload.role or "user")
nginx.conf 中掛載
location /api/ {
    access_by_lua_file /usr/local/openresty/nginx/lua/jwt_auth.lua;
    include /usr/local/openresty/nginx/conf/proxy.conf;
    proxy_pass http://backend_api;
}
實戰二：Redis 快取（content 階段）
lua/cache.lua — 先查快取，miss 才打後端（要搭配 proxy_pass 的版本見下方）
local redis = require "resty.redis"
local cjson = require "cjson"

local red = redis:new()
red:set_timeout(500)

local ok, err = red:connect("127.0.0.1", 6379)
if not ok then
    ngx.log(ngx.ERR, "Redis connect failed: ", err)
    return  -- 連線失敗就直接讓請求繼續到後端
end

local cache_key = "cache:" .. ngx.var.uri
local cached = red:get(cache_key)

if cached and cached ~= ngx.null then
    -- 快取命中，直接回傳
    red:set_keepalive(10000, 100)    -- 用完放回 pool（重要！）
    ngx.header["Content-Type"] = "application/json"
    ngx.header["X-Cache"] = "HIT"
    ngx.say(cached)
    return ngx.exit(200)
end

-- 快取 miss，放回 pool，讓請求繼續到後端
red:set_keepalive(10000, 100)
ngx.header["X-Cache"] = "MISS"
set_keepalive 是必須的：Redis 用完後必須呼叫 red:set_keepalive(timeout_ms, pool_size) 把連線放回 connection pool。忘記這步，高併發下每次請求都新建 TCP 連線，效能急劇下降。
實戰三：Rate Limit（Redis INCR）
lua/rate_limit.lua — 每分鐘限制同一 IP 最多 100 次請求
local redis = require "resty.redis"
local cjson = require "cjson"

local red = redis:new()
red:set_timeout(500)

local ok, err = red:connect("127.0.0.1", 6379)
if not ok then
    ngx.log(ngx.WARN, "Rate limit Redis unavailable: ", err)
    return  -- Redis 掛了就放行（降級策略）
end

-- 用 Client IP 作為 key（若有 Cloudflare 用 http_cf_connecting_ip）
local client_ip = ngx.var.remote_addr
local key = "rate:" .. client_ip

local count, err = red:incr(key)
if count == 1 then
    -- 第一次請求，設定 TTL 60 秒（一個計數窗口）
    red:expire(key, 60)
end

red:set_keepalive(10000, 100)

if count > 100 then
    ngx.status = 429
    ngx.header["Retry-After"] = "60"
    ngx.header["Content-Type"] = "application/json"
    ngx.say(cjson.encode({ error = "Too many requests. Limit: 100/min" }))
    return ngx.exit(429)
end

為什麼需要跳板機？
核心問題：內部的資料庫、後端服務如果直接對公網開放 port，就是巨大的攻擊面。跳板機（Bastion Host）是唯一對公網開放 SSH 的主機，內部所有主機只接受來自跳板機 IP 的連線。

→ SSH :22

203.0.113.10
只開 Port 22 / 80 / 443
OpenResty 也在這裡
→ 私有網路
內部主機群
10.20.35.x
DB / 後端服務
不對公網開放
SSH ProxyJump（現代跳轉方式）
一行指令跳轉
# -J 指定跳板機，直接連到內部主機
ssh -J deploy@203.0.113.10 root@10.20.35.152
~/.ssh/config（推薦：設定一次，永久生效）
# 跳板機設定
Host bastion
    HostName     203.0.113.10
    User         deploy
    IdentityFile ~/.ssh/bastion_key
    Port         22

# 內部 App 主機
Host internal-app
    HostName     10.20.35.152
    User         root
    IdentityFile ~/.ssh/internal_key
    ProxyJump    bastion       # 自動透過跳板機跳轉

# 內部 DB 主機
Host internal-db
    HostName     10.20.35.120
    User         root
    IdentityFile ~/.ssh/internal_key
    ProxyJump    bastion

# 設定好後直接用：
# ssh internal-app
# ssh internal-db
SSH Tunnel（Port Forwarding）
情境：本機無法直接連到內部 MySQL，但可以連到跳板機。透過 SSH Tunnel 讓本機的某個 port 轉發到內部服務。
Local Port Forwarding
# 本機 3307 → 透過跳板機 → 內部 MySQL 3306
# -N 表示不執行任何指令，只做 tunnel
ssh -L 3307:10.20.35.120:3306 deploy@203.0.113.10 -N

# 之後本機 DB 工具直接連 127.0.0.1:3307
mysql -h 127.0.0.1 -P 3307 -u root -p

# 背景執行（加 -f）
ssh -fN -L 3307:10.20.35.120:3306 deploy@203.0.113.10

# 建立 SOCKS5 proxy，本機 1080 port

    listen 443 ssl;
    server_name api.example.com;

    ssl_certificate     /etc/letsencrypt/live/api.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/api.example.com/privkey.pem;
    ssl_protocols       TLSv1.2 TLSv1.3;

    set_real_ip_from 141.101.64.0/18;

    # 健康檢查（不需驗證）

            ngx.say('{"status":"ok"}')
        }

        # rewrite 階段：灰度路由（10% 流量到 v2）
        set $upstream_target "backend_v1";

            local canary = ngx.req.get_headers()["X-Canary"]

                ngx.var.upstream_target = "backend_v2"

        proxy_pass http://$upstream_target;

        proxy_connect_timeout  5s;
        proxy_send_timeout    30s;
        proxy_read_timeout    30s;
        proxy_next_upstream error timeout http_500 http_502;
    }

    location /internal/ {
        allow 10.20.35.0/24;
        deny all;

        proxy_pass http://backend_v1/internal/;

    error_page 500 502 503 504 /50x.html;
    location = /50x.html {
