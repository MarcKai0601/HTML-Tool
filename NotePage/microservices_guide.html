微服務設計原則

每個原則都對應一個現實問題——理解問題才能理解為什麼這樣設計

什麼是微服務？把一個大應用拆成多個小服務，每個服務專注單一業務能力、獨立部署、各自擁有資料庫。核心代價是引入了分散式系統的複雜性——網路不可靠、資料不一致、故障難追蹤。理解微服務就是理解「如何在接受這些代價的前提下，讓拆分帶來的好處最大化」。
Single Responsibility
一個服務只負責一個業務能力（User、Order、Payment 各自獨立）。邊界清晰才能獨立部署、獨立擴展。Conway's Law：你的架構會反映你的團隊結構。
DDD Bounded ContextConway's Law
Database per Service
每個服務擁有自己的 DB，不共享。強制服務間只能透過 API 溝通，避免資料庫層的耦合。代價：跨服務 join 很難，分散式交易需要 Saga pattern。
服務自治Saga pattern 必要
API First
服務之間的介面是唯一契約，先定義 API spec（OpenAPI / Protobuf）再實作。讓不同團隊可以平行開發，也讓前端能提早 mock 資料。
OpenAPI / ProtobufContract testing
Decentralized Governance
各服務可以選擇不同的語言和技術棧——只要 API 契約不變。Order service 可以用 Java，Notification service 可以用 Node.js。
Polyglot persistence技術選型自治
Design for Failure
在分散式系統中，任何服務隨時可能掛掉。設計時必須假設依賴的服務不可用：Circuit Breaker、Retry with backoff、Fallback、Timeout 都是必要的。
Circuit BreakerBulkhead
Observability First
微服務讓 debug 變得極其困難——一個請求可能跨越 10 個服務。Distributed Tracing、集中化 Logging、Metrics 不是選配，是微服務的基礎設施。

資料一致性

Database per Service 讓服務自治，但讓跨服務的資料一致性成為難題

可靠性模式

分散式系統的故障是常態，不是例外。這些模式是讓系統優雅降級的工具

Monolith vs 微服務

微服務不是「更好」，而是「在特定條件下更合適」

最常見的錯誤：一開始就用微服務。微服務的前提是你已經理解業務邊界。Netflix、Amazon 都是先有 Monolith，等業務邊界清晰、團隊夠大之後才拆分。
維度 Monolith Microservices
部署複雜度 低 一次部署全部，簡單直觀 高 各服務獨立部署，需要 K8s / CI/CD 支撐
橫向擴展 受限 只能整體 scale，浪費資源 精準 按服務獨立 scale，Payment 忙就只擴 Payment
故障隔離 差 一個模組 bug 導致整個應用掛掉 佳 Payment 掛掉不影響 Browse / Cart
開發速度（小團隊） 快 無網路呼叫、直接 function call、易 debug 慢 服務間介面、mock、環境建置複雜
開發速度（大團隊） 慢 merge conflict 多、互相阻塞 快 各團隊自治，互不阻塞
資料一致性 強一致 同一 DB，一個 transaction 搞定 最終一致 跨 DB 需要 Saga，複雜度大幅上升
可觀測性 簡單 一個 log file，stack trace 完整 複雜 需要 Distributed Tracing，Trace ID 串接
技術選型 統一 整個應用用同一語言 / 框架 自由 各服務可選最適合的技術棧
網路延遲 無 function call 在 memory 內 有 每次服務間呼叫都是網路 I/O，需 timeout / retry
什麼時候選 Monolith
  • 團隊少於 10 人
  • 業務邊界還不清晰（Startup 初期）
  • 需要快速驗證 MVP
  • 沒有足夠的 DevOps 能力維運分散式系統
  • 資料一致性要求極高（金融核心系統）
什麼時候選微服務
  • 團隊超過 20 人，多個 squad 並行開發
  • 業務邊界清晰，可以定義 Bounded Context
  • 不同模組有極不同的 scaling 需求
  • 有成熟的 CI/CD 和 K8s 維運能力
  • 可以接受 Eventual consistency
Modular Monolith：兩全其美的中間路線
在 Monolith 裡把程式碼按 module 嚴格隔離（各自的 package、各自的 DB schema），不共享直接的程式碼依賴，只透過 interface 溝通。等業務穩定後，可以輕鬆拆成真正的微服務。這是現在許多工程師推薦的起點策略。
