一、什麼是 SSO？

SSO（Single Sign-On，單點登入）讓用戶登入一次，就能使用多個服務，不需要重複輸入帳密。

💡
從 Google 觀察到的：登入 Gmail 之後，打開 YouTube、Drive、Maps 都不需要再登入。這就是 SSO 的用戶體驗。背後的核心是 accounts.google.com 這個統一認證服務。

用戶看到的

進入任一服務
→
導向統一登入頁
→
輸入一次帳密
→
所有服務可用

工程師看到的

Auth Service 負責： 簽發 Token、Refresh Token、撤銷 Token、統一登出 各業務服務 負責： 用 Public Key 本地驗 Token（不需要每個 request 回打 Auth Service） 自己的 RBAC 邏輯

二、兩種設計方案比較

⚠️
討論過程中釐清的關鍵觀念：Library 方案和 SSO 服務方案是兩個不同目標的解法，不能混為一談。

Library 方案（我最初的想法）

把認證邏輯打包成 Maven library，各服務 import 引入。

✓ 無單點故障

✓ 各服務完全獨立

✗ 沒有統一登入體驗

✗ Token 即時撤銷困難

✗ 升級安全邏輯要所有服務重新部署

適合：小型專案、各服務是獨立產品

SSO 獨立服務（業界做法）

Auth Service 獨立部署，各服務用 Public Key 本地驗 Token。

✓ 統一登入體驗

✓ 集中撤銷（Redis 黑名單）

✓ 安全邏輯只改一處

✓ 多實例解決單點故障

✗ 需要維運多個服務

適合：中大型、需要跨服務統一登入

比較項目Library 方案SSO 獨立服務
單點故障✓ 無（library 不會掛）✓ 多實例解決
Token 即時撤銷✗ 困難✓ 集中黑名單
安全邏輯升級✗ 所有服務重新部署✓ 只改 Auth Service
統一登入體驗✗ 各服務各自登入✓ 一次登入全部用
複雜度低較高
真實案例—Auth0、Keycloak、Okta

三、我的專案架構（login_utils 作為 SSO 服務）

四、OAuth2 授權碼流程（核心）

這是 SSO 的核心機制，也是 Google、GitHub 登入背後的原理。

步驟 1：用戶訪問業務服務 工廠系統：「你沒有 Token，去 SSO 登入」 → Redirect 到 SSO，帶上 redirect_uri=工廠系統的callback URL 步驟 2：用戶在 SSO 登入 SSO 驗證帳密成功 → 產生一次性 授權碼（code），有效期很短（10秒） → Redirect 回工廠系統，帶上 code 步驟 3：業務服務後端換 Token 工廠系統後端 → SSO API：「我有 code，換 Token」 SSO 驗證 code 有效 → 回傳 Access Token + Refresh Token 步驟 4：後續請求 工廠系統用 Public Key 本地驗 Access Token → 完全不需要打 SSO！ → 其他已登入的服務也直接可用（Token 格式相同）
✅
為什麼用授權碼（code）而不是直接給 Token？
避免 Token 暴露在瀏覽器 URL 中（瀏覽器歷史記錄、Server Log 都看不到 Token）。code 只能用一次，換完就失效。

五、RS256 vs HS256

HS256（對稱加密）

同一把 secret key 簽發 Token：用 secret 驗證 Token：用 secret 問題： 各服務都要知道 secret → secret 洩漏風險高 → 單一系統可以， 多服務不適合

RS256（非對稱加密）

一對 key：Private + Public 簽發 Token：Private Key（只有 SSO 有） 驗證 Token：Public Key（任何服務都可以有） 好處： 各服務只需要 Public Key → 就算 Public Key 洩漏也沒關係 → 無法偽造 Token → 多服務架構的標準選擇
💡
SSO 對外暴露一個 Public Key API，各服務啟動時下載 Public Key 快取在本地。之後驗 Token 完全不需要網路請求。

六、雙 Token 機制

Access TokenRefresh Token
用途訪問業務 API換新的 Access Token
有效期15 分鐘7 天
存放位置Memory（前端）HttpOnly Cookie（防 XSS）
是否存 DB否（無狀態）是（可以撤銷）
撤銷方式等待自然過期（15分鐘）直接從 DB 刪除
正常流程： 用戶登入 → SSO 發 Access Token（15min）+ Refresh Token（7天） 每次 API 請求帶 Access Token → 業務服務本地驗 Token 過期： Access Token 過期 → 前端自動用 Refresh Token 換新的 Access Token Refresh Token 過期 → 重新登入 強制登出 / 封號： 從 DB 刪除 Refresh Token → 下次刷新失敗 → 用戶被迫重新登入 Access Token 最多再用 15 分鐘（這是可接受的時間差）

九、關鍵學習總結

1️⃣
SSO 和 Library 是不同目標的解法：SSO 解決「用戶體驗（一次登入）」，Library 解決「工程問題（不重複寫代碼）」。我的 login_utils 是 SSO，不是 Library。
2️⃣
本地驗 Token 是關鍵設計：各服務用 Public Key 本地驗，不是每個 request 都回打 SSO。SSO 掛了，已登入用戶照常使用。
3️⃣
授權碼（code）的存在是為了安全：避免 Token 出現在瀏覽器 URL 中，code 只能用一次，換完失效。
4️⃣
雙 Token 解決撤銷問題：Access Token 短期（15分鐘），Refresh Token 存 DB 可即時撤銷。強制登出時刪 Refresh Token，最多 15 分鐘後自動失效。
5️⃣
我觀察到 Google 的現象是對的，推理方向也對，只是共用認證的方式是「獨立服務」而非「Library」。這個思維過程本身在面試時就是加分項。
